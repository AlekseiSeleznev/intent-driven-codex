## TDD Strategy

Apply the project-local `tdd` skill with vertical RED -> GREEN -> REFACTOR slices. Each behavior slice must start from an observable public interface: executable helper scripts, `/opsx:*` prompt files, OpenSpec CLI status/instructions/validation, Git command outcomes in a disposable repository, or documented text checks for prompt/docs-only behavior.

Implementation must not use a horizontal "write all tests, then all code" flow. For each slice, first run the named RED command and capture the expected failure, then implement only enough to make the corresponding GREEN command pass, then refactor only while the GREEN command and surrounding checks still pass.

Use Bash and Python standard-library checks because this repository currently has no dedicated test runner. Do not contact live Claude or read `.secrets.local.env` during these slices; Claude integration is verified through config validation, dry-run/report parsing, and generated effective config checks.

## Mocking / Boundary Policy

- Do not mock internal collaborators owned by this repository; verify behavior through helper script CLIs, prompt/skill files, OpenSpec CLI output, and Git behavior in disposable repositories.
- Use temporary directories and `CODEX_OPENSPEC_SESSION_FILE` or an equivalent test-only environment variable for session-state tests so no real `.codex/session/openspec-session.json` is committed or required.
- Use `--dry-run`, `--validate-config`, and structured parse/config checks for Claude review behavior. Do not invoke live `claude -p`, do not require Claude authentication, and do not read `.secrets.local.env`.
- For budget-exhaustion behavior, use a fixture matching Claude Code JSON result subtype `error_max_budget_usd`; do not burn live budget during required checks.
- Mock only external boundaries that are unavoidable: use disposable Git repositories instead of the real project history for checkpoint helper behavior, and temp JSON configs instead of modifying tracked global defaults.
- Keep all secret-safety tests value-safe: deliberately bad fixtures may use fake placeholder values, but assertions should report only key/path names, not values.
